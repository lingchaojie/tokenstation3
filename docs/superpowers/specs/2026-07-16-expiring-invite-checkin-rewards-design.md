# 可过期邀请与签到奖励余额设计

- 日期：2026-07-16
- 状态：已完成口头设计确认，等待书面复核
- 基线：`dev` @ `b58e9f197`
- 分支：`feat/expiring-reward-credits`
- 涉及模块：affiliate、daily check-in、usage billing、batch image billing、user profile、admin settings、用户导航与余额面板

## 1. 背景

当前邀请奖励只有在受邀方首充达标后发放：邀请方奖励进入 `aff_quota`，需手动转入账户余额；受邀方奖励直接进入 `users.balance`。每日签到奖励也直接进入 `users.balance`，没有独立有效期，无法按来源优先扣费。

本次把邀请与签到奖励改造成统一的可过期奖励余额，并保持用户看到的总余额口径兼容。邀请活动继续由管理员开关控制，并支持“注册立即奖励”和“首充达标奖励”两种模式。

## 2. 目标

1. 邀请奖励支持两种管理员可配置模式：
   - 最低充值阈值为 `0`：邀请码绑定成功时立即发奖。
   - 最低充值阈值大于 `0`：沿用现有首次充值达标后发奖规则。
2. 默认配置改为：受邀方奖励 5、邀请方奖励 10、邀请奖励有效期 7 天、充值阈值 0。
3. 每日签到奖励在领取当日北京时间结束时到期。
4. 请求扣费按“签到 → 邀请 → 订阅 → 账户自身余额”选择一个能够完整承担请求费用的余额层，禁止跨层拆账。
5. 保留订阅余额 fallback 开关：有订阅时，只有用户开启 fallback 才能在订阅不足后使用账户自身余额。
6. 用户总余额数字保持现有口径，并展示签到与邀请奖励余额及到期信息。
7. 管理员可限制每个邀请方最多获得多少次邀请方奖励；受邀方奖励不受该上限影响。
8. 历史可用和冻结返利一次性转入永久有效的账户余额。

## 3. 非目标

- 不增加邀请链接本身的有效期。
- 不增加活动开始/结束时间窗口；继续使用现有总开关。
- 不补发活动关闭期间、立即奖励模式启用前或历史未达标邀请的奖励。
- 不回收已经使用的奖励，也不因首充退款回收已发奖励。
- 不允许余额层之间拆分一笔普通请求或一笔批量图片预冻结。
- 不删除历史邀请、返利和转账审计记录。

## 4. 术语与余额口径

- **签到余额**：来源为每日签到的可过期奖励余额。
- **邀请余额**：邀请方奖励和受邀方奖励的合计可用余额。
- **账户自身余额**：`users.balance` 中扣除当前仍有效、尚未冻结的签到与邀请奖励后的永久/普通余额部分。
- **总余额**：继续使用现有用户可见总余额口径。`users.balance` 保存当前可用总额，`frozen_balance` 保存异步任务冻结额；顶栏总额仍为两者之和。
- **余额层**：签到、邀请、订阅、账户自身余额四个互斥的扣费来源。邀请层内部可由多笔不同到期时间的奖励组成，但仍视为单一余额层。

## 5. 管理员配置

继续复用 `affiliate_enabled` 作为邀请活动总开关，保留现有可配置奖励金额和充值阈值，并新增有效期与获奖次数上限。

| 配置项 | 默认值 | 校验 | 说明 |
| --- | ---: | --- | --- |
| `affiliate_enabled` | `false` | boolean | 邀请活动总开关 |
| `affiliate_first_recharge_threshold` | `0` | 有限数、`>= 0`、最多 8 位小数 | `0` 表示绑定即发；大于 `0` 表示首充模式 |
| `affiliate_inviter_reward` | `10` | 有限数、`>= 0`、最多 8 位小数 | 邀请方奖励 |
| `affiliate_invitee_reward` | `5` | 有限数、`>= 0`、最多 8 位小数 | 受邀方奖励 |
| `affiliate_reward_validity_days` | `7` | 正整数 | 从奖励到账时刻起精确 `N × 24h` |
| `affiliate_inviter_reward_limit` | `0` | 非负整数 | `0` 表示邀请方获奖次数不限 |

`affiliate_rebate_freeze_hours` 不再参与新奖励，后台设置界面移除该字段。旧设置值保留在 settings 表中，以便审计或回滚，但运行时不再读取。

上线迁移对仍等于旧默认值的配置执行条件更新：阈值 `20 → 0`、邀请方奖励 `5 → 10`。管理员已经设置成其他值的配置保持不变。新安装和缺失配置均使用新默认值。

配置变更只影响变更后实际创建的奖励批次；已有奖励的金额和到期时间不改变。

## 6. 邀请活动行为

### 6.1 活动关闭

- 用户侧隐藏“邀请返利”导航入口。
- 直接访问 `/affiliate` 重定向到用户概览。
- 注册页不展示邀请码输入框。
- 带 `aff` 或 `aff_code` 参数访问注册页时，前端通过 `router.replace` 清除邀请参数并停留在普通注册页，不展示活动关闭提示。
- 后端收到手工构造的 `aff_code` 也必须忽略，不绑定邀请关系、不发奖。
- 活动关闭期间发生的注册或首充不补发。

### 6.2 阈值为 0：绑定即发

邀请码校验通过后，在同一数据库事务中：

1. 原子绑定邀请人与受邀人。
2. 增加邀请人的累计绑定人数 `aff_count`。
3. 给受邀方创建受邀奖励批次（金额大于 0 时）。
4. 若邀请方未达到获奖次数上限，原子占用一个名额并创建邀请方奖励批次（金额大于 0 时）。
5. 将邀请关系标记为奖励已处理，防止首充路径再次发奖。

邀请活动失败继续遵循现有注册 fail-open 原则：普通账号注册不因邀请模块故障失败；但邀请绑定、双方奖励和名额占用必须一起回滚，不能出现只绑定或只发一侧的部分状态。错误写入结构化日志供排查。

### 6.3 阈值大于 0：首充达标发奖

保留现有首充规则：

- 只认受邀方第一笔成功充值订单。
- 余额充值到账额必须 `>= affiliate_first_recharge_threshold`。
- 订阅订单继续沿用现有规则，无条件视为达标。
- 第一笔余额充值低于阈值后永久失去资格。
- 活动关闭时完成首充不发奖，重新开启后不补发。
- 重复支付回调不得重复发奖。

支付履约事务锁定受邀关系和邀请方计数后：受邀方按配置获得奖励；邀请方仅在上限允许时获得奖励并占用名额。邀请方达到上限不影响受邀方奖励，也不使支付履约失败。

### 6.4 模式切换与历史绑定

- 阈值从大于 `0` 改为 `0` 时，不立即扫描或补发历史绑定。
- 新绑定按立即奖励模式处理。
- 变更前已经绑定、尚无成功首充的关系保留原“等待首充事件”状态；以后发生首充时按当时配置完成一次性判定，但不会因配置切换本身收到奖励。
- 已有成功首充或已处理过奖励的关系保持终态，不重新开放资格。

### 6.5 邀请方获奖次数上限

上限约束的是“邀请方实际获得邀请方奖励的次数”，不是邀请码绑定人数：

- 所有有效邀请码仍可绑定，`aff_count` 记录全部成功绑定人数。
- 受邀方满足发奖条件后始终可获得受邀方奖励。
- 邀请方最多对前 `N` 个实际发奖事件获得奖励。
- 阈值模式下，只有受邀人首充达标、邀请方奖励实际发放时才占用名额；只绑定未达标不占名额。
- 多个注册或首充并发时，通过邀请方记录行锁/条件更新保证严格不超过上限。
- 邀请方奖励配置为 `0` 时不创建奖励、不占名额。
- 上限降低到已获奖次数以下时立即停止新邀请方奖励；上限调高后可继续发放；改为 `0` 后恢复不限。

`user_affiliates` 新增 `inviter_reward_count` 作为并发安全计数。历史计数从旧邀请返利流水中按 `source_user_id` 去重回填，只统计实际产生过正向邀请方奖励的不同受邀人。

## 7. 邀请关系奖励状态

现有 `user_affiliates.inviter_id` 能表示绑定，但不能区分立即奖励、等待首充、已发奖和首充未达标终止。直接在 `user_affiliates` 增加以下最小状态字段，不另建一对一状态表：

- `reward_mode`：`immediate` / `first_recharge`
- `reward_status`：`pending` / `resolved`
- `reward_resolved_at`
- `reward_source_order_id`（可空）
- 邀请方是否实际获奖、受邀方是否实际获奖

新绑定按当时阈值写入模式。立即奖励事务完成后直接为 `resolved`；首充模式在第一笔成功订单判定后进入 `resolved`，无论达标与否。迁移时：已有奖励或已有成功首充的关系回填为 `resolved`；尚无成功首充的历史绑定回填为 `pending/first_recharge`。

## 8. 统一奖励批次模型

新增 `user_reward_credits` 表，每笔到账奖励对应一个批次：

| 字段 | 说明 |
| --- | --- |
| `id` | bigint 主键 |
| `user_id` | 获奖用户 |
| `credit_type` | `daily_check_in` / `affiliate_inviter` / `affiliate_invitee` |
| `source_key` | 稳定幂等来源键 |
| `original_amount` | 原始奖励金额，decimal(20,8) |
| `remaining_amount` | 当前可用金额，decimal(20,8) |
| `reserved_amount` | 异步任务已冻结金额，decimal(20,8) |
| `granted_at` | 到账时间 |
| `expires_at` | 到期时间，timestamptz |
| `consumed_at` | 全部消费完成时间，可空 |
| `expired_at` | 实际清理时间，可空 |
| `created_at` / `updated_at` | 审计时间 |

约束：

- 金额均不得为负，`remaining_amount + reserved_amount <= original_amount`。
- `(user_id, credit_type, source_key)` 唯一，作为发奖幂等键。
- 活跃查询索引覆盖 `(user_id, credit_type, expires_at)`，并针对 `remaining_amount > 0` 建部分索引。
- 到期清理索引覆盖 `(expires_at, id)`，供分批扫描。

来源键示例：

- 签到：`daily-check-in:<claim_id>`
- 绑定即发邀请方：`affiliate-binding:<invitee_user_id>:inviter`
- 绑定即发受邀方：`affiliate-binding:<invitee_user_id>:invitee`
- 首充邀请方：`affiliate-order:<order_id>:inviter`
- 首充受邀方：`affiliate-order:<order_id>:invitee`

新增奖励事件流水，记录 `grant`、`consume`、`reserve`、`capture`、`release`、`expire`，包含批次、用户、请求/批量任务幂等键、金额和时间，用于对账。普通请求消费与 `usage_billing_dedup` 在同一事务中写入。

批量图片使用奖励余额时，新增冻结分配记录，把一个 hold 映射到一个或多个同类型奖励批次。分配记录必须带唯一 hold 键，保证 reserve/capture/release 幂等。

## 9. 总余额不变量

为保持兼容，`users.balance` 继续是当前可用总额，并包含尚未消费、尚未冻结的奖励余额：

- 发奖：`users.balance += reward`，同时创建奖励批次。
- 普通请求使用奖励：奖励批次减少，`users.balance` 同额减少。
- 奖励到期：未冻结的剩余批次归零，`users.balance` 同额减少。
- 奖励预冻结：批次 `remaining → reserved`，`users.balance` 减少，`frozen_balance` 增加。
- 奖励释放：仅未到期部分 `reserved → remaining`，同步从 frozen 返还到 balance；已到期部分直接失效。
- 奖励 capture：从 reserved 最终消费，减少 frozen，不返还 balance。

账户自身可用余额按下式计算：

```text
account_balance = users.balance
                - active_daily_check_in_remaining
                - active_affiliate_remaining
```

该值允许为负，以兼容现有实际费用超出预估时的欠费记账。管理员调账、退款和普通充值仍更新 `users.balance`，因此等价于调整账户自身余额，不修改奖励批次。

所有奖励余额变动和总余额变动必须在同一个数据库事务中发生。事务完成后统一失效鉴权缓存、余额缓存和奖励汇总缓存。

## 10. 到期语义

### 10.1 签到

签到奖励 `expires_at` 为领取日期在 UTC+8 的次日 `00:00`。例如 2026-07-16 09:30（UTC+8）领取，当日 2026-07-17 00:00（UTC+8）到期。

### 10.2 邀请

邀请奖励按到账时刻精确增加配置天数。例如 2026-07-16 15:30 到账、有效期 7 天，则 2026-07-23 15:30 到期。

### 10.3 逻辑到期与物理清理

所有用户余额读取、请求鉴权、普通扣费和批量图片预冻结均先把 `expires_at <= now` 的批次视为不可用；因此到期语义不依赖后台任务是否准时。

后台清理器使用小批量、行锁和 `SKIP LOCKED` 清理闲置用户的过期批次，并同步扣减 `users.balance`。关键路径仍执行惰性清理，保证清理器延迟或暂时停止时，过期奖励也不能被消费。

到期与消费竞争时，以获得批次行锁的事务为准；同一金额只能消费或过期一次。

## 11. 普通请求扣费

### 11.1 单层全额承担

根据请求实际费用，依次判断：

1. 签到余额合计是否足够。
2. 邀请余额合计是否足够。
3. 订阅是否可完整承担。
4. 账户自身余额是否可完整承担。

高优先级层不足时不得先扣一部分；该层保持原样并尝试下一层。邀请层合计足够时，可在邀请层内部跨多个批次，按 `expires_at ASC, id ASC` 最早到期优先消费，整笔账务来源仍为邀请层。

示例：请求费用 8，签到余额 3、邀请余额 10，则签到余额保持 3 不变，邀请余额扣 8。

### 11.2 订阅 fallback

- 用户没有有效订阅：签到、邀请均不足后直接尝试账户自身余额。
- 用户有有效订阅：签到、邀请均不足后尝试订阅。
- 订阅不足且用户开启 `subscription_balance_fallback_enabled`：再尝试账户自身余额。
- 订阅不足且未开启 fallback：不得使用账户自身余额，保留现有订阅不足/跨界处理语义。
- 奖励余额优先于订阅，不受 fallback 开关影响。

现有订阅窗口重置、无 fallback 的配额跨界、实际费用记账和使用日志语义保持不变；仅在其前增加两个奖励余额候选层，并把最后的余额 guard 改为账户自身余额。

### 11.3 实际费用超预估

请求完成后的实际费用可能高于请求前可判断金额。正常选择禁止跨层拆账；但不能因所有层最终不足而把已经发生的上游调用变成免费。此时沿用现有欠费机制，把无法由奖励或订阅完整承担的实际费用记到账户自身余额，可形成负值，并保留清晰审计记录。

## 12. 批量图片冻结与结算

预冻结金额也必须由单一余额层完整承担，并使用与普通请求相同的签到、邀请、订阅/账户策略中适用于批量图片的余额顺序。当前批量图片只走余额 hold，因此选择顺序为签到 → 邀请 → 账户自身余额。

奖励层预冻结：

1. 锁定同类型活跃批次。
2. 确认该层合计能完整承担 hold。
3. 按最早到期优先把批次金额从 `remaining_amount` 移到 `reserved_amount`。
4. 写入 hold 分配记录，更新 `users.balance` 与 `frozen_balance`。

capture 时按原分配消耗实际金额；多冻结的差额仅在原批次仍未到期时恢复。release 时同理：未到期部分恢复，已到期部分直接失效。所有操作使用现有 batch request ID 作为幂等键。

## 13. 签到改造

`daily_check_in_claims` 继续承担每日唯一领取和历史审计。领取事务改为：

1. 插入当日 claim。
2. 创建 `daily_check_in` 奖励批次，到期时间为下一个 UTC+8 零点。
3. 增加 `users.balance`。
4. 写回 claim 的 `balance_after` 和关联 `reward_credit_id`。该外键对历史 claim 可空，但所有新 claim 必须设置。
5. 提交后失效用户与余额相关缓存。

领取金额仍不计入 `total_recharged`。重复或并发领取由现有唯一约束与奖励来源唯一键共同保证幂等。

## 14. 历史返利迁移

数据库迁移按用户执行一次性转换：

```text
legacy_amount = aff_quota + aff_frozen_quota
users.balance += legacy_amount
aff_quota = 0
aff_frozen_quota = 0
```

- 转入金额成为永久有效的账户自身余额，不创建可过期奖励批次。
- 冻结余额立即释放，不等待原 `frozen_until`。
- `aff_history_quota` 和旧流水保留。
- 插入明确的 `legacy_balance_transfer` 审计记录，带转入后余额快照。
- 迁移文件只执行一次；SQL 还应以旧钱包余额大于 0 为条件，避免工具或人工误执行时重复入账。
- 历史邀请方获奖次数按旧正向 accrue 流水中的不同 `source_user_id` 回填。

新奖励不再进入 `aff_quota`。用户侧移除手动提取入口；原转账 API 保留兼容，但在余额归零后返回 0，管理员历史转账记录继续只读可查。

## 15. API 与 DTO

### 15.1 用户资料汇总

普通用户资料增加：

```json
{
  "reward_balances": {
    "daily_check_in": {
      "amount": 3.0,
      "expires_at": "2026-07-17T00:00:00+08:00"
    },
    "affiliate": {
      "amount": 15.0,
      "earliest_expires_at": "2026-07-23T07:30:00Z",
      "credit_count": 3
    }
  }
}
```

`reward_balances` 及两个子对象始终返回；余额为 0 时返回 `amount: 0`、到期时间 `null`、批次数 `0`。现有 `balance` 和 `frozen_balance` 字段保持兼容。

### 15.2 奖励明细

新增用户只读分页接口，例如：

`GET /api/v1/user/reward-credits?type=affiliate&status=active&page=1&page_size=20`

返回每笔当前剩余金额、奖励角色（邀请方/受邀方）、到账时间和到期时间。只允许读取当前用户记录，默认只返回活跃批次，按最早到期排序。

### 15.3 邀请详情

用户邀请详情增加：

- `first_recharge_threshold`
- `inviter_reward`
- `invitee_reward`
- `reward_validity_days`
- `inviter_reward_limit`
- `inviter_reward_count`
- `inviter_reward_limit_reached`

只有 `limit > 0 && count >= limit` 时前端展示上限提示。展示值为配置上限，例如历史 count 已超过新下调上限时仍显示 `3/3`，不显示 `5/3`。

### 15.4 管理端

管理员 settings DTO 增加有效天数与上限字段，移除 freeze-hours UI。保留现有邀请奖励记录路由和 API 路径，但把数据源改为“统一奖励事件 + 历史返利流水”的兼容查询，并标注奖励模式与双方结果；这样立即奖励、首充奖励和旧记录都能在同一页面审计。旧转账记录页继续只读保留。

## 16. 前端交互

### 16.1 动态邀请文案

阈值为 0：

> 邀请立刻获得返现。好友注册并成功绑定邀请码后，你获得 {inviterReward}，好友获得 {inviteeReward}，奖励 {validityDays} 天内有效。

阈值大于 0 时保留当前首充达标文案，动态插入阈值、邀请方奖励和受邀方奖励。订阅无条件达标的现有说明继续保留。

达到邀请方获奖上限时才显示：

> 邀请奖励已达 {limit}/{limit}。好友仍可获得受邀奖励，你将不再获得新的邀请方奖励。

分享链接继续可用；未达到上限或上限为 0 时不显示进度提示。

### 16.2 总余额奖励明细（已选方案 C）

默认紧凑显示：

- `签到余额 ¥X · 到期时间`
- `邀请余额 ¥X · 最早到期时间 · N 笔明细`

点击邀请明细后按到期时间展开分页列表，并标注“邀请好友奖励”或“受邀奖励”。对应余额为 0 时不渲染该 bullet。顶栏余额浮层与用户概览余额卡复用同一个奖励汇总组件，保证桌面端和移动端均可查看。

### 16.3 活动关闭

公开 settings 中的 `affiliate_enabled` 控制用户菜单和注册输入。注册页在 public settings 已加载前不渲染邀请码字段，避免关闭状态下闪现。路由守卫处理 `/affiliate`，注册页处理邀请查询参数；服务端仍是最终授权边界。

### 16.4 后台设置

邀请设置卡展示：总开关、最低充值阈值、双方奖励金额、有效天数和邀请方获奖次数上限。阈值输入旁明确说明 `0 = 注册绑定后立即发奖`，上限旁明确说明 `0 = 不限制，达到上限后受邀方仍可获奖`。

## 17. 缓存与一致性

- 奖励发放、消费、到期、reserve/capture/release 和历史迁移后失效用户鉴权缓存及余额缓存。
- 余额资格缓存改为结构化快照，包含总余额、签到可用合计、邀请可用合计及最近到期时间；请求前资格判断使用该快照，缓存 miss 时回源数据库。
- 到期时间必须参与缓存 TTL，缓存不得越过最近奖励到期点继续命中。
- 扣费最终决策始终在数据库事务内重新校验，缓存只用于请求前资格判断。

## 18. 错误处理

- 非法邀请码、自邀和重复绑定保持现有错误语义。
- 活动关闭时邀请码字段静默忽略，不阻断普通注册。
- 奖励唯一键冲突视为已处理的幂等成功，不重复增加总余额。
- 邀请方达到上限是正常业务分支，不返回错误。
- 到期清理器单批失败只记录并重试，不影响请求路径的惰性到期判断。
- 奖励批次与 `users.balance` 更新任何一侧失败，整个事务回滚。
- 缓存失效失败记录 warning，不回滚已经提交的账务事务。

## 19. 测试策略

### 19.1 后端业务测试

- 阈值 0：注册绑定后双方按配置发奖，金额与到期时间正确。
- 阈值大于 0：余额首充边界、订阅达标、首充不足永久失格、重复回调。
- 活动关闭：不绑定、不发奖、不补发。
- 模式切换：历史绑定不因改成 0 立即补发，新绑定立即发。
- 任一侧奖励金额为 0；邀请方为 0 时不占上限。
- 9 个并发受邀事件、上限 3：9 个受邀方全部获奖，邀请方严格 3 次。
- 上限降低、提高、改为 0；历史计数回填。

### 19.2 奖励账本与到期测试

- 发奖、唯一来源幂等、余额同步和事务回滚。
- 签到奖励 UTC+8 次日零点边界。
- 邀请奖励精确 `N × 24h` 边界。
- 到期与消费并发，只能处理一次。
- 清理器延迟时，查询和扣费仍不能使用过期余额。
- 管理员调账不修改奖励批次，账户自身余额计算正确。

### 19.3 普通扣费测试

- 签到层足够、签到不足而邀请足够、邀请不足而订阅足够。
- 高优先级层不足时保持原金额，不发生部分扣款。
- 邀请层跨多个批次，最早到期优先。
- 有/无订阅，fallback 开/关的组合。
- 账户自身余额 guard 不误用奖励余额。
- 实际费用超预估时保留欠费审计，不产生免费调用。
- `usage_billing_dedup` 重放不重复消费奖励。

### 19.4 批量图片测试

- 单层完整 reserve；不足时不部分冻结。
- 多邀请批次最早到期分配。
- capture 返还未用金额。
- release 恢复未到期批次。
- hold 期间到期的差额不恢复。
- reserve/capture/release 重放幂等。

### 19.5 迁移测试

- `aff_quota + aff_frozen_quota` 一次性进入永久账户余额。
- 冻结余额立即释放，旧钱包归零，历史累计与流水保留。
- 条件默认配置更新不覆盖管理员自定义值。
- 历史获奖次数按不同受邀人去重回填。
- 重复执行保护不产生二次入账。

### 19.6 前端测试

- 活动关闭时菜单、路由、邀请码输入和邀请链接参数清理。
- 阈值 0 与大于 0 的动态文案。
- 达到上限时才显示 `{limit}/{limit}`。
- 总余额数字不变，零余额 bullet 隐藏。
- 邀请汇总显示最早到期与批次数；点击后按需分页加载明细。
- 管理设置字段加载、校验、保存和默认值。

## 20. 验收标准

1. 新安装默认阈值 0、邀请方 10、受邀方 5、有效期 7 天、获奖上限 0。
2. 活动开启且阈值 0 时，新用户绑定邀请码后双方立即获得奖励；活动关闭时用户端无入口、注册页无邀请码字段、后端不绑定。
3. 阈值大于 0 时保持当前首充达标语义，并根据配置动态展示文案。
4. 邀请方达到获奖上限后不再获得新奖励，但后续受邀方仍正常获得奖励；并发场景不超发。
5. 签到奖励在领取当日 UTC+8 次日零点不可用；邀请奖励在到账后配置天数到期。
6. 普通请求从签到、邀请、订阅、账户自身余额中选择第一个能完整承担费用的层，不跨层拆账。
7. 有订阅时账户 fallback 只在用户开关开启后生效；无订阅时可直接使用账户自身余额。
8. 邀请层内部按最早到期优先消耗，批量图片冻结/结算也保持来源与到期一致性。
9. 用户总余额数字保持兼容，并按方案 C 展示签到与邀请奖励汇总及按需明细。
10. 历史可用及冻结返利一次性进入永久账户余额，旧钱包归零且不重复入账。
11. 后端全量测试、lint、前端 lint/typecheck/相关组件测试及生成代码检查通过。
