# 管理员使用记录排除用户筛选设计

## 目标

在管理员“使用记录”页面保留现有单用户正向筛选，并新增一个可搜索、可多选的“排除用户”筛选。被选用户的记录不得出现在页面明细、统计、图表、排行、错误记录和导出结果中。

## 范围

排除条件作用于：

- 使用记录明细及其分页总数
- 顶部统计卡
- 模型、分组、端点分布和 Token 趋势图
- 用户排行
- 错误记录列表
- Excel 导出

使用记录清理任务不继承排除条件。清理接口继续只接受现有正向过滤条件，避免把“排除少数用户”误解为“删除其余全部用户记录”。

## 交互设计

现有“用户”单选搜索栏保持不变，其旁新增“排除用户”栏：

- 输入邮箱片段或用户 ID 时，复用现有管理员用户搜索接口及其匹配规则。
- 搜索结果包含活跃用户和已删除用户，并沿用已删除用户标识。
- 点击搜索结果会加入排除列表；已选用户显示为可单独移除的标签。
- 已选用户不会重复出现或重复加入。
- 最多选择 100 个排除用户；达到上限后不再允许加入，并显示提示。
- 当前由“用户”栏正向选中的用户不能加入排除列表。
- 如果管理员随后在“用户”栏选择了已被排除的用户，该用户会自动从排除列表移除，确保条件不冲突。
- 页面“重置”会清空排除列表。

## API 契约

相关管理端查询新增可选查询参数：

```text
exclude_user_ids=12,34,56
```

参数规则：

- 使用逗号分隔的正整数用户 ID。
- 服务端去重并按升序归一化。
- 空值等同于未设置。
- 非数字、零、负数或超过 100 个唯一 ID 时返回 `400 Bad Request`。
- 所有缓存键必须包含归一化后的排除用户 ID，避免不同筛选条件共享缓存结果。

前端类型内部使用 `number[]`，仅在请求边界编码为逗号分隔字符串。这样组件状态保持类型安全，也不会依赖 HTTP 客户端对数组参数的序列化格式。

## 后端数据流

管理员 handler 统一解析 `exclude_user_ids`，再传入用量查询过滤结构和错误记录过滤结构。仓储查询在原有条件上增加排除谓词。

对于 `usage_logs.user_id` 或错误记录 `user_id` 为空的历史记录，排除条件不得把它们一并过滤。因此 SQL 语义为：

```sql
(user_id IS NULL OR NOT (user_id = ANY($n)))
```

而不是直接使用 `user_id NOT IN (...)`。正向 `user_id` 与排除列表同时存在时仍按 AND 语义执行；正常前端交互会避免冲突，服务端则保持确定、可组合的查询语义。

以下查询路径必须接入同一排除条件：

- 管理员使用记录列表与统计
- 统计内的端点、上游端点和端点路径聚合
- Dashboard snapshot v2 的趋势和分组聚合
- 独立模型统计
- 用户排行聚合
- Ops 错误记录列表

Excel 导出复用使用记录列表参数，因此不单独实现过滤逻辑。

## 错误处理

- 用户搜索失败时保持现有静默空结果行为，不改变已经选中的排除用户。
- 任一查询收到非法 `exclude_user_ids` 时返回明确的 400 响应，不执行宽泛查询。
- 页面数据请求失败沿用现有各区域的错误提示和旧数据保留策略。
- 选择达到上限时只阻止新增选择，不清除已有筛选。

## 测试策略

前端测试覆盖：

- 搜索、加入多个用户、去重、逐个移除和重置。
- 已删除用户标识及活跃用户优先排序。
- 正向用户与排除列表的冲突消解。
- 明细、统计、模型图、snapshot v2、用户排行、错误记录及导出请求携带相同排除 ID。
- 达到 100 个用户后的限制提示。

后端测试覆盖：

- 参数解析、归一化、去重、非法输入和 100 个上限。
- 列表与汇总 SQL 使用参数化数组排除用户。
- 被排除用户消失，其他用户及 `NULL user_id` 记录保留。
- 模型、趋势、分组、端点和用户排行聚合口径一致。
- Ops 错误记录应用相同排除语义。
- Dashboard 缓存键包含排除列表，顺序不同但集合相同时生成相同键。

## 非目标

- 不新增持久化的管理员排除名单。
- 不将排除规则扩展到 API Key、账号或分组。
- 不改变用户搜索接口的匹配算法。
- 不让清理任务支持反向删除范围。
