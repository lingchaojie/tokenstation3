# 每日签到领 Token 活动设计

## 1. 背景与目标

为普通用户增加一个限时的“签到领Token”活动。活动仅在管理员启用且当前时间位于配置的活动窗口内时显示。普通用户按 UTC+8 自然日每天可签到一次，签到成功后立即获得管理员配置的账户余额，默认奖励为 10 USD。

本次同时调整两处导航文案与结构：

- 普通用户左侧菜单中的“仪表盘”改名为“概览”，保留 `/dashboard` 路由与管理员仪表盘文案。
- 管理员左侧菜单中的“邀请返利”父菜单改名为“福利中心”，保留已有邀请记录、返利记录和提取记录，并新增“签到配置”子菜单。

## 2. 范围

### 2.1 包含

- 管理员配置活动启用状态、开始时间、持续天数和每日奖励金额。
- 普通用户查看当日签到状态并领取奖励。
- UTC+8 活动窗口与自然日边界计算。
- 原子、幂等的余额入账与签到审计记录。
- 领取后鉴权缓存和余额缓存失效。
- 活动有效期内的普通用户菜单显隐。
- 中英文界面文案、后端行为测试和前端交互测试。

### 2.2 不包含

- 连续签到、连续签到额外奖励或断签规则。
- 补签、代签、排行榜或签到提醒。
- 用户签到历史页或管理员签到流水查询页。
- 按用户、用户组或注册时间配置不同奖励。
- 对既有邀请返利业务规则、邀请记录、返利记录和提取记录的改造。

## 3. 已确认的产品规则

1. 只有普通用户可以领取，管理员不能通过接口领取。
2. 活动配置包含独立启用开关；只有开关开启且时间窗口有效时活动才处于进行中。
3. 活动开始时间可精确到分钟，管理员界面固定按 UTC+8 输入和展示。
4. 持续时间按正整数天配置，结束时刻等于 `开始时间 + 持续天数 × 24 小时`。
5. 活动窗口使用左闭右开区间：`start_at <= now < end_at`。
6. 每个 UTC+8 自然日只能领取一次。漏签不能补领。
7. 活动若在某日零点以后开始，用户当日仍可领取；跨过下一个 UTC+8 零点后可再次领取。
8. 每日奖励进入账户余额，默认值为 10 USD，且不计入 `total_recharged`。
9. 修改奖励金额只影响修改后的领取，历史领取金额保持不变。
10. 修改活动开始时间视为新一轮活动。持续天数或奖励金额的修改不创建新活动身份。

## 4. 管理员界面

### 4.1 导航结构

管理员左侧菜单调整为：

```text
福利中心
├── 邀请记录
├── 返利记录
├── 提取记录
└── 签到配置
```

“福利中心”不再受邀请返利开关控制，否则邀请返利关闭时管理员无法进入签到配置。它继续遵循现有简易模式隐藏规则。普通用户侧的“邀请返利”菜单不改名。

新增前端路由 `/admin/affiliates/check-in`，渲染独立签到配置页。已有 `/admin/affiliates/*` 路由和默认重定向保持不变。

### 4.2 签到配置页

页面包含：

- “启用签到活动”开关。
- UTC+8 活动开始日期时间。
- 持续天数，只接受正整数。
- 每日奖励余额，只接受大于 0、最多 8 位小数的 USD 金额，默认 10。
- 根据开始时间和持续天数实时计算的只读结束时间。
- 当前活动状态：已关闭、未开始、进行中或已结束。

关闭活动时允许保留其余配置。启用活动时，后端必须验证开始时间存在、持续天数大于 0、奖励金额有限且大于 0。前端校验用于即时反馈，后端校验是最终依据。

### 4.3 管理端 API

- `GET /api/v1/admin/check-in/config`：返回完整配置、计算后的结束时间和当前状态。
- `PUT /api/v1/admin/check-in/config`：整体保存配置并返回规范化后的配置和状态。

接口复用现有管理员鉴权，不向普通用户开放。

## 5. 普通用户界面

### 5.1 导航

- 普通用户左侧菜单的 `/dashboard` 标签使用新文案“概览”。管理员 `/admin/dashboard` 以及管理员“我的账户”中的个人仪表盘文案保持不变。
- 活动进行中时，在“概览”之后显示“签到领Token”菜单，链接到 `/check-in`。
- 活动关闭、未开始或已结束时隐藏该菜单。
- 菜单显隐可根据公开活动窗口和一个轻量前端时钟计算，使保持打开的页面能在活动开始和结束边界更新；这只是展示逻辑，不能决定领取资格。

### 5.2 签到页

`/check-in` 页面展示：

- 每日奖励金额。
- UTC+8 活动开始和结束时间。
- 今日状态：可领取或已领取。
- 下一个 UTC+8 零点刷新时间。
- “立即签到领取”按钮。

领取成功后页面立即显示实际到账金额与领取后余额，按钮切换为已领取状态，并刷新当前用户余额。页面保持打开并跨过 UTC+8 零点时重新查询状态，恢复可领取状态。

通过地址直接访问失效活动页面时，不重定向；页面显示活动未进行并禁用领取，以便正确处理菜单状态尚未刷新、活动刚结束或管理员刚关闭活动的情况。

### 5.3 用户端 API

- `GET /api/v1/user/check-in/status`：返回活动状态、窗口、每日奖励、今日是否已领取、今日领取信息和下次 UTC+8 零点。
- `POST /api/v1/user/check-in/claim`：执行当日领取，返回实际奖励、领取后余额、签到日期和领取时间。

两个接口都要求登录。领取接口还必须验证当前角色为普通用户。

## 6. 配置模型与时间语义

配置沿用现有 settings 存储与 SettingService 读取模式，新增四个设置键：

- `daily_check_in_enabled`，默认 `false`。
- `daily_check_in_start_at`，默认空字符串，保存规范化的 RFC3339 UTC 时间。
- `daily_check_in_duration_days`，默认 `0`。
- `daily_check_in_reward_amount`，默认 `10.00000000`。

公开设置只暴露菜单显隐所需的启用状态、开始时间和计算后的结束时间，不暴露管理端写接口。用户状态与领取资格始终由后端基于最新配置重新计算。

时间计算固定使用 UTC+8 的无夏令时位置，不能依赖服务器本地时区。后端把 `now` 转换到 UTC+8 后生成 `YYYY-MM-DD` 签到日期，并计算下一个 UTC+8 零点。测试通过注入或传入时钟覆盖边界，不使用真实等待。

## 7. 数据模型

新增 `daily_check_in_claims` 表，字段如下：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | bigint | 主键 |
| `user_id` | bigint | 领取用户 |
| `activity_start_at` | timestamptz | 本轮活动身份与开始时间快照 |
| `check_in_date` | date | UTC+8 签到自然日 |
| `reward_amount` | decimal(20,8) | 实际到账金额快照 |
| `balance_after` | decimal(20,8) | 到账后余额快照 |
| `claimed_at` | timestamptz | 实际领取时间 |
| `created_at` | timestamptz | 创建时间 |

约束与索引：

- 外键 `user_id -> users.id`。
- 唯一约束 `(user_id, activity_start_at, check_in_date)`。
- 查询索引 `(user_id, activity_start_at, check_in_date)` 和 `claimed_at`。

记录为账务审计依据，不提供业务删除接口。若以后需要清理，只能通过独立的数据保留策略处理。

## 8. 后端组件边界

### 8.1 CheckInService

职责：

- 读取并验证活动配置。
- 计算活动状态、UTC+8 签到日期和下次重置时间。
- 查询用户今日签到状态。
- 编排领取事务。
- 事务成功后失效鉴权和余额缓存。

服务层使用可替换时钟，确保时间边界可测试。

### 8.2 CheckInRepository

职责：

- 查询指定用户、活动和签到日期的领取记录。
- 在单个数据库事务中创建领取记录并增加用户余额。
- 返回实际奖励和领取后余额。

仓储层不能调用现有 `UserRepository.UpdateBalance`，因为该方法会把正向金额加入 `total_recharged`。签到入账必须只更新 `users.balance` 和 `updated_at`。

### 8.3 Handler

用户 Handler 从鉴权上下文读取用户 ID 和角色，输出稳定 DTO。管理员 Handler 只负责请求绑定与错误映射，配置规范化和业务验证由服务层完成。

## 9. 领取事务与并发

领取流程：

1. 读取最新配置并用服务端当前时间验证活动正在进行。
2. 计算 UTC+8 `check_in_date`，固定本次 `reward_amount` 和 `activity_start_at`。
3. 开启数据库事务。
4. 插入签到记录；唯一冲突映射为“今日已领取”。
5. 仅增加 `users.balance`，读取新余额。
6. 把 `balance_after` 写回签到记录。
7. 提交事务。
8. 提交成功后失效用户鉴权缓存和余额缓存。

若插入、余额更新或余额快照更新任一步失败，整个事务回滚，不允许出现“有记录没到账”或“到账但没有记录”。并发请求由数据库唯一约束裁决，不能使用先查后写作为幂等保证。

## 10. 错误处理

服务层提供可识别错误：

- 活动未进行：关闭、未开始、已结束或配置无效。
- 今日已领取。
- 仅普通用户可领取。
- 配置字段非法。
- 用户不存在。

前端收到“今日已领取”时重新拉取状态并展示已领取，而不是保留失败按钮。活动未进行时刷新活动状态并禁用按钮。数据库或缓存错误使用现有统一错误展示；缓存失效失败记录日志，但不撤销已经提交的奖励事务。

## 11. 测试策略

### 11.1 后端

- 活动关闭、未开始、开始瞬间、结束前和结束瞬间的窗口判定。
- UTC+8 23:59 到次日 00:00 的签到日期与重置时间。
- 活动中途开始、跨日和结束时刻不是零点的组合。
- 默认 10 USD 与管理员自定义奖励。
- 单日首次领取成功，余额增加但 `total_recharged` 不变。
- 重复领取与并发领取仅有一次入账。
- 插入后余额更新失败时事务完整回滚。
- 奖励金额修改只影响后续记录；开始时间修改形成新活动身份。
- 管理员领取被拒绝。
- 成功领取后的鉴权缓存和余额缓存失效。
- 管理端配置校验、规范化和权限。
- API 契约中新增字段和路由保持稳定。

### 11.2 前端

- 普通用户菜单只在活动窗口内显示，并位于“概览”之后。
- 普通用户显示“概览”，管理员仪表盘文案不变。
- 管理员父菜单显示“福利中心”，包含四个预期子项，不再依赖邀请返利开关。
- 签到配置表单校验、结束时间预览、状态展示与保存。
- 签到页初始状态、领取成功、重复领取、活动结束和跨零点刷新。
- 领取成功后用户余额刷新。
- 中英文关键文案存在且可解析。

## 12. 兼容性与上线

- 新设置键具有安全默认值，升级后活动默认关闭，不会自动向用户发放余额。
- 新表通过项目现有数据库迁移流程创建；应用代码发布前不得假设表已存在。
- 既有邀请返利配置、用户邀请入口和三类管理员记录路由不变。
- `/dashboard` 路径、路由名称和仪表盘数据接口不改，降低前端导航回归风险。
- 不需要后台定时任务；签到资格在状态查询和领取请求时按当前时间计算。

## 13. 验收标准

1. 管理员能在“福利中心 → 签到配置”保存合法活动，非法配置不能启用。
2. 活动只有在启用且当前时间位于窗口内时出现在普通用户菜单。
3. 普通用户在同一活动的每个 UTC+8 自然日最多成功领取一次。
4. 领取成功后账户余额立即增加配置金额，累计充值金额不变。
5. 重复点击和并发请求不会重复入账。
6. 活动关闭、未开始或结束后，接口不发放奖励。
7. 普通用户菜单显示“概览”，管理员仪表盘仍显示原文案。
8. 管理员父菜单显示“福利中心”，已有邀请返利记录功能继续可用。
