## Range-Specific Local Protection Baseline

All implementation and verification for this pinned range must preserve these
TokenStation3 behaviors:

- Keep `region` as the IDC/OIDC authentication region and retain the
  independently configured per-account `api_region` for Q runtime routing.
- Keep access-token cache identity scoped by `account.ID`; do not regress to a
  credential-derived key that can collide across accounts.
- Keep the persisted stable machine ID and its current derivation rules so
  refresh-token rotation does not change the KIRO runtime identity.
- Keep validated account-level custom upstream headers, and preserve the
  profile ARN contract: Q chat uses body `profileArn` without the profile
  header, Q MCP uses `x-amzn-kiro-profile-arn`, and KRS uses both body and
  header with the existing fallback behavior.
- Keep direct KIRO gateway routing local behavior; do not replace it with a
  reverted or superseded reference-fork route.
- Keep mixed scheduling behavior for direct KIRO and Anthropic-compatible relay
  accounts, including the separation between direct accounts and relay accounts
  that set `base_url`.
- Keep local capture/archive redaction boundaries, authoritative usage and
  credit accounting, cache emulation, and final usage propagation across
  adapters.

Every later implementation commit and its focused verification must check the
applicable items in this baseline. Advancing the KIRO reference is blocked if a
selected change weakens any item without an explicit user decision.

## General upstream feature decisions

- Integrate endpoint mode `auto` as an opt-in capability. Preserve `q` as the
  default, preserve account-level runtime configuration, and keep API Key
  accounts on Q.
- Exclude the four approved scope-expansion groups: upstream billing-rate
  introspection/probe/scheduling; OpenAI/Codex reliability fixes; image-input
  pricing, usage, and OAuth image usage; and Grok passive image intent and key
  templates.
- Later product-specific, non-general, operations, and test-convenience
  features may be excluded from this KIRO sync at the category level only
  after the archive documents both what each feature does and why excluding it
  is compatible with the pinned KIRO scope. A category label alone is not a
  sufficient rationale.
- KIRO core behavior, a real conflict with protected local behavior, or
  ambiguous product semantics is never eligible for automatic exclusion. The
  auditor must report the capability and obtain a user decision before
  assigning its disposition.

These decisions apply only to the pinned audit range and do not replace the
normal Wei-Shaw upstream-sync process.
