## 合并原则

沿用既定规则：
- 上游原生、我们**没特意改过**的功能 → 直接跟随上游。
- 我们已有**特制行为**的区域 → 以本地为准，只合入能兼容这些行为的上游增量。
- 不确定会改变线上语义的功能，不借合并默认打开。
- **不是单纯合代码**：逐处 code review 功能影响,绝不无脑 `--theirs`。

## 方法：两跳合并（Two-Hop）

本次特殊点：上游在我们上次同步后的 2 天窗口内（2026-07-08）做了一次**「纯移动拆分」大重构**——把 6 个巨型 service/repo 文件掏空、拆成几十个小文件，而我们恰好也往这些巨型文件里塞了大量定制代码（capture / kiro tap）。直接 `git merge` 会因「上游删了 6000 行 + 我们散点改动」导致 diff3 失准、**静默错合 + 重复符号**。

因此拆成两跳，把「功能变更」和「文件搬家」分开解决：

| 跳 | commit | 内容 | 冲突性质 |
|---|---|---|---|
| Hop 1 | `1d4f6db20` | 合入重构前边界 `f68f3b86c`（= `d754be0d8^`），79 个功能 commit | 普通功能冲突（文件仍是单体） |
| Hop 2 | `e073f63b6` | 合入 `ws/main`（含 6 大单体拆分 + 75 commit） | 纯结构搬迁：整取上游拆分 + 把我们的 tap 重贴到新文件 |
| 修正 | `f22722a35` | go.mod → 1.26.5 对齐 CI；删用户侧死键 `payment.planCard.peakRate` | 复审驱动 |
| 卫生 | `b7d0bb886` | `.gitignore` 屏蔽 graphify-out / .superpowers | — |

> 该模式已验证有效，供后续同步复用：先合重构前边界，再合结构重构。

## 关键取舍与偏离上游的决定

1. **affiliate 用我们的奖励模型,不用上游返利率模型**（跨 4 层：schema / service / handler / 前端）。这是产品决定,后续同步继续保持;别让上游的 rate/cap 字段回流。
2. **migration 重编号**：我们与上游在 159-166 撞号。上游占 159-172（batch-image + grok-video）,我们 7 个撞号迁移重编到 **173-179**。runner 按完整文件名记账,重编只为顺序整洁 + 排在上游之后;已迁移库上会以新文件名幂等重跑（`IF NOT EXISTS`,安全）。
3. **Go 1.26.5**：采纳上游安全升级。CI 用 `go-version-file: backend/go.mod` + `grep go1.26.5`,故 go.mod 必须为 1.26.5。（注：本次本地验证在 1.26.4 上完成,仅版本指令差异、代码中性。）
4. **i18n 模块化**：上游删平铺 `en.ts/zh.ts` 改域模块树。我们新增的 webchat 键归入新建 `en|zh/webchat.ts`,其余叶子键分发到对应域模块,保持 en/zh 键对齐 + 通过冲突守卫测试。
5. **`openai_gateway_responses_chat_fallback.go`**：上游把请求构建抽成共享 `sendCCUpstreamRequest`,已吸收我们的 CC-raw header 白名单 + 传输层 failover,故直接采纳上游版（不再保留旧内联代码）。
